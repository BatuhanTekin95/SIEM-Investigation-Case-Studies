# Active Directory Lateral Movement Investigation (Splunk)

## Introduction

After investigating phishing attacks, account compromise, persistence mechanisms, and web shell activity in previous case studies, I wanted to focus on another critical stage of the attack lifecycle: lateral movement.

In many real-world intrusions, compromising a single workstation is only the beginning. Once attackers gain an initial foothold, they often attempt to move throughout the environment in search of higher-value systems such as file servers, database servers, and domain controllers. To achieve this, they frequently abuse legitimate administrative protocols and tools including SMB, PsExec, PowerShell, and RDP.

One of the challenges defenders face is that these same technologies are used every day by system administrators, making it difficult to distinguish normal administrative activity from malicious behavior.

In this investigation, I used Splunk to analyze log data generated during a simulated Active Directory compromise. The objective was to identify discovery activity, trace lateral movement between systems, investigate the use of SMB, PsExec, and RDP, and reconstruct the attacker's path through the environment.

Throughout the investigation, I correlated events from multiple log sources, examined authentication activity, reviewed process creation events, and mapped observed techniques to the MITRE ATT&CK framework to better understand the attacker's actions and objectives.

## PowerShell-Based AD Enumeration


After identifying several discovery commands through Sysmon Event ID 1 logs, I expanded the investigation to include PowerShell Script Block Logging (Event ID 4104).

While Sysmon provides visibility into process creation events, Script Block Logging captures the actual PowerShell commands executed during a session. This allows defenders to observe attacker activity in greater detail, including Active Directory enumeration performed through PowerShell cmdlets.

To identify PowerShell-based discovery activity, I searched for common Active Directory enumeration commands such as `Get-ADUser`, `Get-ADGroupMember`, and `Get-ADComputer`.

<img width="1905" height="632" alt="Ekran görüntüsü 2026-06-14 212351" src="https://github.com/user-attachments/assets/358eff93-b972-417c-8d81-a78b2184a3ec" />

> PowerShell Script Block Logging (Event ID 4104) revealed the use of Active Directory enumeration cmdlets, including `Get-ADUser`, `Get-ADGroupMember`, and `Get-ADComputer`. These commands enabled the attacker to identify domain users, privileged group memberships, and domain-joined systems, providing valuable intelligence for planning lateral movement within the environment.

One of the most significant findings was the use of the following PowerShell command to enumerate domain user accounts:

`Import-Module ActiveDirectory; Get-ADUser -Filter * -Properties MemberOf | Select-Object Name, SamAccountName`

This command allowed the attacker to retrieve domain user information and group membership data directly from Active Directory, providing valuable intelligence for selecting targets and planning subsequent lateral movement activity.

## SMB-Based Lateral Movement Investigation

After identifying Active Directory discovery activity, the next step was to determine whether the attacker used the gathered information to move laterally across the environment.

SMB administrative shares such as `ADMIN$`, `C$`, and `IPC$` are commonly used by administrators for remote management. However, attackers frequently abuse these shares to transfer files, execute commands, and establish access to additional systems after obtaining valid credentials.

To investigate potential SMB-based lateral movement, I reviewed Windows Security Event ID 5140 logs to identify access to administrative shares and determine which accounts and systems were involved.

> Windows Security Event ID 5140 logs revealed repeated access to the ADMIN$ administrative share across multiple systems using the account `luke.sullivan`. The activity originated from source IP `10.5.50.12` and targeted several hosts within a short period of time. Access to administrative shares on multiple systems is commonly associated with SMB-based lateral movement and warranted further investigation.

The investigation identified several ADMIN$ share access events occurring within minutes of each other. The same account accessed multiple hosts, including THM-DEV-WS, THM-SHR-SRV, and THM-SQL-SRV, from a single source IP address.

At this stage, the account `luke.sullivan` appeared to be responsible for the activity. However, administrative credentials can be stolen or abused by another user. To determine who was actually initiating the connections, I needed to trace the activity back to the source system and identify the user operating behind the scenes.

While administrative share access can be legitimate, the pattern observed here differed from normal administrative behavior. The rapid access to multiple systems suggested that the credentials were being used to facilitate lateral movement across the environment.

## Tracing the Activity Back to the Source Host

> Event ID 4624 analysis revealed that source IP 10.5.50.12 was associated with the machine account THM-MKT-WS$. Since machine accounts end with the $ character in Active Directory, this identified THM-MKT-WS as the workstation initiating the suspicious SMB connections.

The discovery significantly changed the direction of the investigation. Previous analysis showed that the administrative account luke.sullivan was accessing multiple ADMIN$ shares across the environment. However, the source host was not the administrator's normal workstation.

Instead, the activity originated from THM-MKT-WS, a marketing workstation that had already been identified as the initially compromised system during earlier phases of the investigation.

This strongly suggested that the attacker had obtained privileged credentials and was leveraging them from the compromised workstation to authenticate to remote systems and expand access throughout the Active Directory environment.

Having confirmed that stolen administrative credentials were being used from the compromised workstation to access remote ADMIN$ shares, the next step was to determine whether the attacker progressed beyond file share access and began executing commands on remote systems through PsExec.


## PsExec-Based Lateral Movement Investigation

While SMB administrative share access confirmed that the attacker was authenticating to remote systems, file share access alone does not necessarily indicate remote command execution.

To determine whether the attacker progressed beyond simple SMB access, I investigated artifacts associated with PsExec, a legitimate Windows administration tool frequently abused by attackers for lateral movement.

PsExec allows an operator to execute commands remotely by combining SMB administrative share access with temporary service installation on the target host. Because of this behavior, PsExec generates a distinct set of artifacts across Windows Security, System, and Sysmon logs that can be used to reconstruct attacker activity.

The objective of this phase was to identify the destination systems targeted through PsExec, determine which commands were executed remotely, and trace the activity back to the originating workstation.

> PsExec combines SMB administrative share access with temporary service installation to execute commands remotely. This process generates artifacts across Security, System, and Sysmon logs that can be used to trace lateral movement activity.

## Identifying PsExec Service Installation

After establishing SMB-based access to multiple systems, I investigated whether the attacker progressed to remote command execution using PsExec.

One of the most reliable indicators of PsExec activity is Windows System Event ID 7045, which records the installation of new services on a host. Because PsExec deploys a temporary service on the target system to execute commands remotely, service creation events often provide the first clear evidence of PsExec-based lateral movement.

<img width="1893" height="338" alt="Ekran görüntüsü 2026-06-14 225815" src="https://github.com/user-attachments/assets/e3f68b83-d435-46df-8292-82cfda943e5f" />

> Windows System Event ID 7045 revealed the creation of the PSEXESVC service on THM-SQL-SRV. The service was configured as a demand-start user mode service and executed under the LocalSystem account, a common artifact generated by PsExec during remote command execution.

The investigation identified two separate installations of the PSEXESVC service on THM-SQL-SRV within a short period of time. The presence of this service strongly suggested that the attacker had moved beyond simple SMB share access and was actively executing commands on the target server through PsExec.

While Event ID 7045 confirmed that PsExec had been used on the destination host, it did not reveal which commands were executed. The next step was to analyze process creation events generated by the PsExec service to determine the attacker's actions on the target system.

## Investigating Commands Executed Through PsExec

After confirming the installation of the PSEXESVC service on THM-SQL-SRV, the next objective was to determine what actions the attacker performed on the target system.

Service creation alone confirms the use of PsExec but does not reveal the commands executed through it. To identify the attacker's activity, I analyzed Sysmon Event ID 1 process creation logs and filtered for processes spawned by `PSEXESVC.exe`.

Because PsExec executes commands through its temporary service, any child processes created by `PSEXESVC.exe` provide direct visibility into the attacker's actions on the destination host.

<img width="1889" height="370" alt="Ekran görüntüsü 2026-06-14 230148" src="https://github.com/user-attachments/assets/cedbe7e7-24d3-48de-b362-49289b34894b" />

> Sysmon Event ID 1 logs on THM-SQL-SRV revealed commands executed directly through the PSEXESVC service. The ParentImage field showed that both commands were spawned by PSEXESVC.exe, confirming remote execution through PsExec.

The first command executed through PsExec was:

`cmd /c "hostname & whoami & ipconfig"`

This command was likely used for host validation and situational awareness. By collecting the hostname, current user context, and network configuration, the attacker could verify successful access to the target system and gather information useful for further lateral movement.

Shortly afterward, the attacker executed:

`cmd /c "net localgroup administrators"`

This command enumerated local administrator group membership on THM-SQL-SRV, allowing the attacker to identify privileged accounts and assess opportunities for additional privilege escalation or lateral movement.

These commands demonstrated that the attacker was performing post-compromise reconnaissance on THM-SQL-SRV after successfully establishing remote execution through PsExec. The activity moved beyond simple authentication and provided clear evidence of interactive lateral movement within the environment.

## Named Pipe Artifacts (Sysmon Event ID 17)

After identifying commands executed through the PSEXESVC service, I investigated named pipe creation events generated by PsExec.

PsExec relies on named pipes to exchange input, output, and error streams between the source and destination systems. These artifacts are recorded by Sysmon Event ID 17 and often provide valuable evidence of remote execution activity.

To identify PsExec-related pipes, I searched for pipe creation events associated with PSEXESVC.exe.


<img width="1898" height="533" alt="Ekran görüntüsü 2026-06-14 230222" src="https://github.com/user-attachments/assets/226f8c13-b67a-42aa-8ea4-b95c3a8f65a2" />

> Sysmon Event ID 17 logs revealed multiple named pipes created by PSEXESVC.exe on THM-SQL-SRV. The pipe names included stdin, stdout, and stderr channels used by PsExec to communicate between the source and destination systems.

The pipe names contained the hostname THM-MKT-WS, revealing the system from which PsExec was executed.

Examples included:

- PSEXESVC-THM-MKT-WS-4544-stdin
- PSEXESVC-THM-MKT-WS-4544-stdout
- PSEXESVC-THM-MKT-WS-4544-stderr

These artifacts provided additional confirmation that the attacker was operating from the compromised workstation THM-MKT-WS while executing commands remotely on THM-SQL-SRV.

The named pipe artifacts independently validated earlier findings obtained from Event ID 4624, Event ID 5140, and Sysmon Event ID 1. Multiple log sources now consistently identified THM-MKT-WS as the origin of the PsExec activity, strengthening confidence in the lateral movement timeline.

## Tracing PsExec Activity Through Named Pipe Artifacts

After identifying the PsExec service installation and the commands executed on THM-SQL-SRV, I reviewed Sysmon Event ID 17 and Windows Event ID 5145 logs to identify additional PsExec artifacts.

PsExec creates named pipes such as stdin, stdout, and stderr to communicate between the source and destination systems. These artifacts often remain visible even when attackers attempt to disguise their activity.

Event ID 5145 logs revealed multiple PsExec-related named pipe artifacts, including PSEXESVC-THM-MKT-WS-4544-stdin, stdout, and stderr. The pipe names contained the hostname THM-MKT-WS, directly identifying the compromised marketing workstation as the source of the remote execution activity.

The same events showed that the privileged account luke.sullivan accessed the ADMIN$ and IPC$ shares from source IP 10.5.50.12, confirming that PsExec was being used to execute commands remotely on THM-SQL-SRV.

By correlating Event IDs 7045, 5145, Sysmon Event ID 1, and Sysmon Event ID 17, the investigation established a complete PsExec execution chain from the compromised workstation THM-MKT-WS to the target server THM-SQL-SRV.


## Tracing PsExec Execution Back to the Source Host

After identifying PsExec service creation on THM-SQL-SRV and reviewing the commands executed through PSEXESVC.exe, I pivoted back to the suspected source workstation to validate where the activity originated.

To accomplish this, I reviewed Sysmon Event ID 1 process creation logs on THM-MKT-WS and searched for executions of PsExec.exe.

This command was likely used to verify successful access to the target system and collect basic host information.

While the source and destination systems had now been identified, additional artifacts remained available for validation. To further confirm PsExec activity and identify evidence of remote execution channels, I investigated named pipe creation events generated during the attack.

## RDP-Based Lateral Movement Investigation

After identifying SMB-based lateral movement and PsExec execution across the environment, the next step was to determine whether the attacker established interactive remote sessions to move deeper into the network.

Unlike SMB shares or PsExec service creation, Remote Desktop Protocol (RDP) provides attackers with a full interactive desktop session. Successful RDP logons generate Windows Security Event ID 4624 with Logon Type 10, making them a valuable artifact for tracing lateral movement.

To investigate potential RDP activity, I correlated Security Event ID 4624 logon events with Sysmon process creation logs to identify the source systems involved and reconstruct the attacker's movement path through the environment.

## Identifying Suspicious Activity on the Domain Controller

After completing the investigation of SMB and PsExec activity, the next step was to determine whether the attacker had established interactive remote access to additional systems within the environment.

To begin, I reviewed Sysmon Event ID 1 process creation logs on the Domain Controller (THM-DC) and searched for common discovery commands frequently used during post-exploitation.

## Conclusion

This investigation demonstrated how an attacker leveraged valid credentials and multiple lateral movement techniques to progress through the environment and ultimately gain access to the Domain Controller.

The analysis began with suspicious SMB administrative share access, where Windows Security Event ID 5140 logs revealed repeated connections to ADMIN$ shares across multiple systems. Further investigation showed that the attacker was operating from a compromised workstation while abusing the credentials of a legitimate administrative account.

Additional analysis uncovered the use of PsExec for remote command execution. System Event ID 7045 and Sysmon Event ID 1 logs confirmed the creation of the PSEXESVC service and revealed the commands executed on remote hosts. These artifacts provided clear evidence of attacker-controlled remote administration and system enumeration activities.

The investigation then identified RDP-based lateral movement. By correlating Security Event ID 4624 logon events with Sysmon process creation logs, it was possible to reconstruct a complete RDP chain from the compromised workstation to an intermediate server and ultimately to the Domain Controller. This activity demonstrated how the attacker pivoted through multiple systems while leveraging valid administrative credentials to expand access within the environment.

By combining evidence from Windows Security logs and Sysmon telemetry, the full attack path was successfully reconstructed. The investigation highlighted the importance of monitoring administrative shares, service creation events, process execution activity, and remote logon events to detect and investigate lateral movement within Active Directory environments.

Overall, the findings demonstrated a coordinated intrusion involving SMB-based lateral movement, PsExec remote execution, and RDP chaining techniques, providing a practical example of how attackers move between systems after obtaining valid credentials.
